# Registrations module

Owns registration containers and attendee participation records. It must support individual, household, group, worker, and imported paths without requiring a club relationship.

Public form submissions create a normal `Registration`, an ordered attendee party, and a one-to-one `PublicRegistrationSubmission`. The submission stores the exact published version, sanitized registration/attendee responses, request hash/idempotency key, and immutable server pricing snapshot. Capacity-aware answers create normalized `RegistrationCapacityReservation` rows in the same serializable transaction. Anonymous submissions may reuse an existing person by normalized email but never overwrite that person’s canonical profile; event-specific contact details remain in registration and attendee snapshots.

Explicit lifecycle operations own cancellation, reactivation, waitlist entry, manual promotion, and automatic earliest-fitting promotion. They release or restore capacity atomically, preserve totals, maintain queue positions, write audit history, and queue transactional notices. Staff edits cannot bypass status transitions. Expiring hash-only private links expose contact-only self-service plus read-only attendees, status, payment history, and Square checkout.

Authorized staff can use a two-step review to transfer an entire registration to a new account holder or substitute one attendee in place. Both operations require a client UUID, run in a serializable transaction, persist immutable before/after snapshots, and return the exact stored response on a same-input retry. A transfer changes only the account holder and contact snapshot: the registration, party, answers, order, totals, payments, refunds, promo redemption, capacity reservations, and waitlist state keep their existing identities and values. It revokes active management links in the transaction and issues the replacement contact's deterministic private access only after commit.

An attendee substitution updates the existing attendee row's person and profile snapshot without changing its ID, position, attendee type, form responses, pricing, or capacity claims. A checked-in attendee cannot be substituted until staff undo the active check-in, and the replacement cannot duplicate another person already in the party. Both workflows write a separate audit record and queue versioned notices without storing or logging raw access tokens. They remain staff-only; the private public workflow does not expose transfer, substitution, cancellation, or refund controls.
